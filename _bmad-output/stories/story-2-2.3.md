# Story 2.3: Warden Approval Dashboard

## Description
As a **warden**,
I want an approved leave to automatically generate a verifiable QR-code gate pass,
So that students can present a cryptographically signed pass at the gate for quick, secure verification.

## Status: Complete

## Acceptance Criteria

**AC-1:** Given a leave transitions to APPROVED (via Story 2.2 approve flow), when `createGatePass(leave)` is called by the approve service, then a GatePass document is created with: qrToken (JWT signed with QR_SECRET, not JWT_SECRET), passCode (6-digit random numeric string), jti (UUID), status ACTIVE, expiresAt matching the leave's endDate, and lastGateState defaulting to 'IN'

**AC-2:** Given the QR token is generated by `createGatePass`, when it is decoded with QR_SECRET, then the payload contains only `{ leaveRequestId, jti, exp }` -- no studentId or other PII is embedded in the token

**AC-3:** Given QR_SECRET is a separate env var from JWT_SECRET, when JWT_SECRET is rotated for auth key management, then existing gate passes remain valid (their qrTokens were signed with QR_SECRET)

**AC-4:** Given the gate pass has expiresAt set to the leave's endDate, when the endDate passes, then the gate pass is naturally expired (queries filter by `expiresAt > now`)

**AC-5:** Given a leave is approved, when the gate pass is created, then the leaveId field has a unique index ensuring one gate pass per leave (duplicate creation throws)

**AC-6:** Given the GatePass status constants are defined, when any code imports `GatePassStatus`, then the enum contains exactly ACTIVE, USED, EXPIRED, and CANCELLED states

**AC-7:** Given a STUDENT is authenticated, when they GET `/api/gate-passes/active`, then the server returns the active gate pass for that student (status ACTIVE, expiresAt > now), or `{ gatePass: null }` if none exists

**AC-8:** Given a non-STUDENT role is authenticated, when they GET `/api/gate-passes/active`, then the server returns 403 FORBIDDEN

## Technical Context
- **QR token signing:** `jwt.sign({ leaveRequestId, jti }, env.QR_SECRET, { expiresIn })` -- QR_SECRET is separate from JWT_SECRET to isolate gate pass verification from auth token verification
- **passCode generation:** `randomInt(100000, 999999)` from `node:crypto` for cryptographically secure 6-digit codes
- **jti:** `randomUUID()` from `node:crypto` for unique token identifiers
- **expiresIn calculation:** `Math.max(1, Math.floor((leave.endDate.getTime() - Date.now()) / 1000))` seconds -- minimum 1 second to avoid negative expiry
- **GatePass model:** Unique index on `leaveId` ensures one pass per leave; compound index on `{ studentId: 1, status: 1 }` for active pass queries
- **Route mounting:** Gate pass routes mounted at `/api/gate-passes` in `server/src/app.ts`
- **Integration point:** `createGatePass(leave)` is called by `approveLeave` in `server/src/services/leave.service.ts` (Story 2.2)
